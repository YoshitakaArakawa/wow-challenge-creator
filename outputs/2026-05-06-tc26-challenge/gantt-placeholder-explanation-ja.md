# Gantt Placeholder パターン解説（Extreme 難易度）

TC26 Challenge の「Calendar (Extreme)」シートで使われている、**同時刻に並走するセッションを横にずらして並べる**ための表計算パターンを解説します。

---

## 1. 何をしたいのか

- 行（Y 軸）: 時刻
- 列（X 軸）: 日付ペインの中で、**重なり合うセッションをレーン 0, 1, 2... と横にずらす**
- マーク: セッションごとに縦の帯（Line）

各セッション（= 各 `id`）に**横オフセット整数**を 1 つ割り当てたい。これを `2_Gantt_Placeholder_1` という名前の**プレースホルダー（仮の連続軸）**として列に置くのが狙い。

## 2. 1 本の式で書けない理由

Tableau の計算モデル上、このオフセットを 1 式にまとめることは事実上できません。

| 制約 | 困ること |
|---|---|
| **A. 再帰累計が必要** | 「重なるたびに +1」は `PREVIOUS_VALUE` を使う再帰でしか書けない |
| **B. 隣接マーク比較が必要** | 「直前セッションの終了時刻 ≥ 現セッション開始時刻」は `LOOKUP(…, -1)` 必須 |
| **C. ネストごとに Addressing を変えたい** | 「セッション境界を見る計算」と「時間軸を走る計算」で表計算の向きが異なる |

特に **C** が決定的で、1 つの式に対して Addressing は 1 組しか指定できません。そのため**ネストした表計算**に分解し、ネストごとに Addressing を切り替えて迂回します。

## 3. マークの構造（前提）

- データ: `sessions`（1 行 = 1 セッション）× `minutes`（1 行 = セッション中の 1 分）を結合
- マーク LOD: **(id × 分)**
- 日付ペインで絞られるので、**パーティション = Date、アドレス空間 = 全 (id, 分) マーク**
- `id` の並び順は computed-sort で `0_Sorter = STR(date) + STR(start_time) + STR(duration)` = **開始時刻昇順**に固定

## 4. 各計算の役割

### ① `0_Time_Axis`（次元・行レベル）
```
DATEADD('minute', [Minute], [start_time])
```
セッション開始時刻＋経過分のタイムスタンプ。後段の表計算で `LOOKUP` 比較される基礎値。

---

### ② `0_INDEX = INDEX()`
- **Addressing: `id` のみ**（パーティション = 時刻）
- **役割**: 各時刻スライスで「この時刻に居るセッション群」を id 順に 1, 2, 3... と番号付け。最小開始時刻のセッションだけが `②=1` になる。

---

### ③ `0_Is_Gantt_Start = INDEX() = 1`
- **Addressing: `0_Time_Axis` のみ**（パーティション = id）
- **役割**: 「このセッションの最初の分」を TRUE にする。id ごとに時間軸を走らせるので、`minute=0` の行だけが TRUE = **セッション開始の瞬間**。

② と ③ は**直交する Addressing**。② は "時刻固定で id 方向"、③ は "id 固定で時刻方向"。両方を組み合わせることで「パーティション全体の先頭マーク」をピンポイントで特定できる。

---

### ④ `0_Is_Gantt_Start_Overlap`
```
MIN([0_Time_Axis]) <= LOOKUP(MIN([0_Time_Axis]), -1)
```
- **Addressing: `id` → `0_Time_Axis`**（両方を順に走査）
- **役割**: 「直前マークと比べて時刻が逆戻り or 同じか」＝**セッション間の重なり検出器**。

Addressing を (id, time) にするとマークは次の順で一直線に並ぶ：

```
A min0, A min1, ..., A minN, B min0, B min1, ..., B minM, C min0, ...
```

- 同一 id 内: 時刻は +1 分ずつ進む → FALSE
- id 境界（A の最後 → B の最初）: B の開始時刻 ≤ A の終了時刻なら TRUE = **B と A は重なる**

`LOOKUP` を含むので行レベルでは書けず、表計算にする必要がある。

---

### ⑤ `1_Delta_1 = IF ④ THEN 1 ELSE 0`
- **Addressing: 表（横）/ Rows**（ネストした ④ の Addressing が優先されるため実質無関係）
- **役割**: ブール → 数値への橋渡し。分けている理由は**可読性＋再利用**。増分値を変えたくなったらここだけ触ればよい。

---

### ⑥ `2_Gantt_Placeholder_1`
```
IF [0_Is_Gantt_Start] AND [0_INDEX] = 1
THEN 0
ELSE [1_Delta_1] + PREVIOUS_VALUE(0)
END
```
- **Addressing: `id` → `0_Time_Axis`**（パーティション = Date）
- **役割**: 再帰累計の本体。
  - 先頭マーク: 0 で初期化
  - それ以外: **前マークの値 +（重なっていれば 1、なければ 0）**

初期化条件 `③ AND ② = 1` は「先頭セッションの先頭分」= パーティション全体でただ 1 マークのみ真。再帰累計の起点。

## 5. 動作トレース

例: A (9:00-10:00), B (9:30-10:30), C (11:00-12:00), D (11:15-12:15)

| マーク | ④ | ⑤ | ⑥（オフセット） |
|---|---|---|---|
| A min0（先頭）| — | — | **0**（初期化） |
| A min1..60 | FALSE | 0 | 0 |
| B min0（A と重なる）| TRUE | 1 | 0 + 1 = **1** |
| B min1..60 | FALSE | 0 | 1 |
| C min0（B と重ならない）| FALSE | 0 | 1 |
| C min1..60 | FALSE | 0 | 1 |
| D min0（C と重なる）| TRUE | 1 | 1 + 1 = **2** |
| D min1..60 | FALSE | 0 | 2 |

**同一セッション内の全マークは同じ X 座標**に乗るため、線がまっすぐ縦に引け、重なるセッションだけ右にずれる。

## 6. 表計算ダイアログでの設定

⑥を選択 → 「表計算の編集」を開くと下記のネスト構造になる：

```
2_Gantt_Placeholder_1
├── 次を使用して計算: 特定のディメンション
│     順序: id → 0_Time_Axis（両方 Field 順）
│
├─[ネスト] 0_Is_Gantt_Start          → 0_Time_Axis のみ
├─[ネスト] 0_INDEX                   → id のみ
├─[ネスト] 1_Delta_1                 → 表（横）
└─[ネスト] 0_Is_Gantt_Start_Overlap  → id → 0_Time_Axis
```

**ネストごとに Addressing を切り替える**のがこのパターンの肝。全員を同じ Addressing に統一すると、③の `INDEX()=1` が全マーク中の先頭 1 つにしか立たず、「各セッションの開始」を拾えなくなって初期化ロジックが壊れる。

## 7. ビュー側の仕上げ

- **列**: `ATTR([Date]) * 2_Gantt_Placeholder_1` — 日付でペインを区切り、中を横オフセットで展開
- **行**: `0_Time_Axis`（連続、`reverse=true` で上から下に時間進行）
- **マーク**: Line、詳細 = `id`、色 = `ATTR(AI Session Flag)`
- プレースホルダー軸は `display=false` でラベル非表示、`range-type=independent` で日付ごとに独立
- `mark-transparency=180`、Size 小さめで「細長い帯」として Gantt バー化

## 8. なぜ 6 段構成なのか（まとめ）

1. 行レベルでは書けない（`LOOKUP` / `PREVIOUS_VALUE` が必要）→ 表計算化
2. 再帰には「今回の増分」と「初期化条件」が必要 → 増分側（④⑤）と初期化側（②③）に分離
3. 初期化条件は「セッション開始」と「パーティション開始」を同時判定する必要があり、前者は時刻 Addressing、後者は id Addressing → ②③ に分離
4. 増分検出は隣接 id 境界を見る必要あり → ④を独立
5. ⑤ は可読性のための薄いラッパ
6. ⑥ で再帰累計に集約

「1 式に Addressing は 1 組」という Tableau の構造的制約を、**ネスト計算＋ネストごとの個別 Addressing** で迂回しているのがこのパターンの本質です。
